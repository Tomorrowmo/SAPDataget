# 报告清单"8506 / 返回 0 / Excel 200"根因与修复

## 现象（生产，真实 SAP）
用户在生产点"报告清单"：提示**共 8506 条**、**返回 0 条**、但 **Excel 里有 200 条**。三者自相矛盾。

## 取证（tools/sap_probe.py 在生产实跑）
对 `ZBW_QUERY_LIST_SRV/LtResultSet` 不带 `$top` 直接 GET：
- HTTP 200，`application/atom+xml`。
- **`<m:count>` = 无**；**单次响应实际 entry 条数 = 8506**；**无分页 next 链接**。
  → 该服务**一次性返回全部 8506 行**（所有用户的报告），不分页、不带 count。
- 字段名：**`Uname`**(报告归属用户) / `Compid`(报告ID) / `Txtlg`(描述)。样本
  `Uname='SESA10052', Compid='ZHKI_M01_Q001', Txtlg='HK Report: SPA USAGE'`。
- 数据按 `Uname` 排序，前段全是 `SESA10052` 的报告；登录用户是 `sesa550670`。

## 根因
报告清单这条链路（server.py `_run_report_list_shortcut` 的 live 分支）：
1. `rows_all = _parse_report_list_xml(...)` → 全部 8506 行。
2. `row_count_total = total_count(None) or len(rows_all)` → **8506**。
3. `rows = rows_all[:top_n]`，top_n 默认 200 → 取**前 200 行**（恰好全是 `SESA10052` 的，不是你的）。
4. 之后按登录用户(`Uname`)过滤这 200 行 → **0**（你的报告在 8506 里靠后，不在前 200）。
而 Excel 写的是过滤前的 200 行（或前端只在预览处过滤）→ 出现"Excel 200 / 返回 0"。

**一句话：先 `[:200]` 再按用户过滤——顺序反了。** 全量已在内存，应**先过滤、再截断**。

## 修复（本批）
`_run_report_list_shortcut` live 分支：**先按登录用户过滤全量 `rows_all`(归属字段 `Uname`，
大小写不敏感)，再 `[:top_n]`**；总数改用"你名下的真实条数"，answer 标注"（你 X 名下）"。
离线模拟验证：旧逻辑返回 0 → 新逻辑返回该用户的 35 条。mock 路径无 `Uname`，自动 no-op，不受影响。

## 待定优化（需再确认）
该服务每次返回 8506 行 XML（约 MB 级），较浪费。若它**支持 `$filter`**，可改为请求时带
`$filter=Uname eq '登录用户'` 让 SAP 只返回你的报告（payload 大幅减小）。是否支持需看探针
第 3 节（`execute_query(filter=Uname eq '...')` 是否成功）——待用户回传后再加服务端注入，
并把通用 build_excel 路径的按用户过滤也一并改为服务端 `$filter`。
