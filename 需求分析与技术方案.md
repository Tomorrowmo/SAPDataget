# 第一部分　产品分析

## 1. 目标

### 1.1 商业目标
| # | 目标 | 衡量指标 |
|---|---|---|
| B1 | **降低业务取数成本** | 业务部门人均月取数时长从 X 小时 → < 30 分钟 |
| B2 | **释放 BW 团队产能** | 临时取数工单从 N 单/月 → < 30%，BW 团队转向建模与治理 |
| B3 | **沉淀报表资产** | 90 天内沉淀 ≥ 30 个常用模板（Skills），覆盖 ≥ 70% 高频请求 |
| B4 | **数据访问可治理** | 每次取数全程可审计；敏感字段集中管控 |

### 1.2 用户目标
| 角色 | 「我想……」 | 「为了……」 |
|---|---|---|
| 业务用户 | 用大白话说一句就能拿到数据 | 不必学 BEx、不必排队等 IT |
| 业务用户 | 拿到的是 Excel，能直接编辑、二次整理、邮件给老板 | 不必再人工把屏幕表格抄到 Excel |
| BW 管理员 | 把高频报表「封装成模板」、一次维护多人受益 | 不再重复回答同一个问题 |
| 高管/老板 | 助理 5 分钟内拿到数据 | 不耽误决策 |

### 1.3 产品目标 (3 个月)
1. **覆盖率**：常见取数请求 ≥ 70% 一次问到答（不需要管理员介入）。
2. **首次出 Excel 时长**：P50 < 20s，P95 < 60s（不含 BW 后端慢查询）。
3. **模板库**：上线时 ≥ 5 个种子模板；3 个月内累计 ≥ 30 个。
4. **业务 NPS** ≥ 40。
5. **零安全事件**：无越权数据访问、无敏感字段泄露。

## 3. 定位

### 3.1 一句话定位

> **给业务用户的 SAP BW 取数副驾驶 —— 用大白话提需求，拿到能直接用的 Excel。**

### 3.2 是什么（产品边界 · 正向）

| 维度 | 是 |
|---|---|
| 形态 | 浏览器 SPA + 后端智能体 |
| 输入 | 自然语言 + 模板参数表单 |
| 输出 | Excel 文件（数据 + 格式 + 可选图表） |
| 数据源 | SAP BW 7.5 既有 OData 服务 |
| 核心引擎 | 可配置 LLM 适配层（LiteLLM）——支持 Claude / Qwen / DeepSeek / GPT / 本地 Ollama，详见 §7 |
| 知识载体 | Skills 模板库（管理员可扩展） |

### 3.3 不是什么（产品边界 · 反向）

- **不是 BI 工具**：不替代 Tableau / Power BI / SAC 做交互式仪表盘
- **不是聊天机器人**：不闲聊、不做通用数学、不做代码生成
- **不是数据建模工具**：不创建 / 修改 InfoCube、DSO、MultiProvider
- **不是数据中台**：不做跨系统 ETL、不做主数据治理
- **不是事务系统**：只读，不回写 SAP

### 3.4 与替代品的差异

| 替代品 | 业务用户当前痛点 | 本产品如何更好 |
|---|---|---|
| **BEx Analyzer / Query Designer** | 需要培训才能用、装客户端、要懂 InfoObject | 浏览器即用、自然语言 |
| **Analysis for Office (AfO)** | 同上，并且大量企业已逐步退役 | 同上 |
| **找 IT/BW 排队** | 几天到一周才能拿到一份数据 | 30 秒内 |
| **SAP Analytics Cloud (SAC)** | 部署成本高、需要 SAC 许可证、面向仪表盘而非临时取数 | 仅需 LLM API + 现有 BW Gateway，临时取数体验更轻 |
| **导出 BW 全表手动 Excel 透视** | 工作量大、易出错、行数受限 | 智能体一步到位 |

### 3.5 与 v0.1 设计的关键差异

| 维度 | v0.1 | v0.2 (当前) |
|---|---|---|
| 核心交互 | 像 ChatGPT 一样滚动文本 | 围绕「一次取数任务」组织，产出 Excel |
| 输出 | 屏幕上的 Markdown 表格 + 可下载 CSV | **Excel 是一等输出**，CSV 是退路 |
| 知识沉淀 | 元数据缓存 | **Skills 模板系统**（业务知识+模板+查询逻辑的封装） |
| 用户入口 | 单一对话框 | 三入口：**模板画廊** + **自由提问** + **历史重跑** |
| 模板维护者 | （未定义） | BW 管理员（非开发） |

### 4.2 典型场景（Job Stories 写法）

**场景 1 · 月度销售简报（模板驱动 · 高频）**
> *当* 每月 1 号上班 *我* 是华东大区经理，*我想要* 上个月华东各办事处的销售额、毛利率、同比环比，
> *这样我就能* 在 9 点的晨会用 Excel 一页讲清状况。
>
> 期望交互：点开「月度销售大区简报」模板 → 选「华东」「2026-05」→ 下载 Excel。

**场景 2 · 临时数据问询（对话驱动 · 中频）**
> *当* 老板突然问「A 客户最近 3 个月订单量怎么样」，*我* 是客户经理，
> *我想要* 30 秒内回答，*这样我就能* 不让老板等。
>
> 期望交互：自由提问 → 智能体识别需要 ZBW_ORDER_SRV → 自动取数 → 出 Excel + 一句话总结。

**场景 3 · 对比分析（多步组合 · 中频）**
> *当* 季末结账时 *我* 是财务分析师，*我想要* 三个季度的费用对比 + 与预算的差异，
> *这样我就能* 写月报。
>
> 期望交互：提问 → 智能体追问「哪三个季度？只看费用还是含收入？」→ 用户答 → 出多 sheet Excel + 差异透视。

**场景 4 · 异常排查（探索 · 低频）**
> *当* 库存看板显示某 SKU 突然激增，*我* 是供应链计划员，*我想要* 查这个 SKU 过去 12 周的入库 / 出库 / 调拨明细，
> *这样我就能* 定位异常单据。
>
> 期望交互：模板 + 改参数 → 出明细 Excel。

**场景 5 · 模板维护（管理员视角 · 低频）**
> *当* 业务同事反复问「能不能加一个『付款逾期客户清单』模板」，*我* 是 BW 管理员，
> *我想要* 不写代码就能新增模板，*这样我就能* 把这类高频请求一次性解决。
>
> 期望交互：登录后台 → 新建模板 → 填提示词 / 选服务 / 上传 Excel 模板 / 试运行 / 发布。

### 4.3 反例场景（**不要在产品里支持**）

- "帮我画一个销售漏斗的实时大屏" → 推用户去 SAC
- "把这份 Excel 数据导回 BW" → 推用户去 IT
- "给我所有客户的明细 80 万行" → 拒绝，提示加筛选；产品定位是分析数据不是 dump 数据
- "今天天气怎么样" → 严格拒绝，不闲聊

### 4.4 场景 → 入口映射

| 场景 | 入口 | 占比预估 |
|---|---|---|
| 月度/周度固定报表 | 模板画廊 | 50% |
| 临时问询 | 自由对话 | 30% |
| 历史报表参数改一改 | 历史重跑 | 15% |
| 探索 / 排查 | 自由对话 + 多轮追问 | 5% |

# 第二部分　技术方案

## 5. 设计原则

1. **Excel-first**：每次任务的终态是 Excel 文件；屏幕展示只是预览。
2. **Skills 即知识**：业务知识（哪个服务、哪些字段、怎么聚合、Excel 长啥样）下沉到模板，**模板由非程序员维护**。
3. **LLM 是粘合剂、不是数据搬运工**：BW 数据**绝大部分不经过 LLM**（LLM 只看摘要 + 前 N 行），完整数据走后端直通 Excel。
4. **任务式交互**：以"一次取数任务"为最小单元（status: 新建 → 收参 → 运行 → 完成/失败），不是无终态对话。
5. **B/S、零安装**：浏览器即用；后端单 Windows Server 起步，容器化部署。
6. **BW 权限不绕开**：用每个用户自己的 BW 凭据连 BW，Analysis Authorization 自动生效。

## 7. LLM 抽象层（关键基础设施）

### 7.1 为什么不绑死一家 LLM

| 驱动 | 说明 |
|---|---|
| **网络可达性** | Claude / GPT 需公网出口或代理；Qwen (Dashscope) / DeepSeek **国内直连**，某些客户网络环境只能用国内模型 |
| **数据合规** | 涉密数据要求模型权重在本地（私有化部署 Qwen2.5-72B / DeepSeek-V3 / Llama 等） |
| **成本** | DeepSeek-V3 输入 ≈ ¥1/M token、Claude Opus 4.7 ≈ ¥36/M token，**差 36 倍**；批量取数场景一年差几万 |
| **能力分层** | 难问题用强模型（Claude/GPT-4o），简单参数抽取用快模型（qwen-plus / deepseek-chat），按场景路由能省 ≥ 70% 成本 |
| **避免供应商锁定** | 一家提价、限流、终止服务、出口管制时，**30 分钟内切换** |
| **业务用户中文体验** | 国内模型（Qwen / DeepSeek）对中文行业术语、SAP 字段名理解更稳 |

**结论：LLM 是基础设施，必须抽象。**

### 7.2 适配方案：LiteLLM

[LiteLLM](https://github.com/BerriAI/litellm) 把所有主流 LLM 厂商的 API 统一成 **OpenAI 兼容格式**
（`messages` + `tools` + `tool_calls`）。后端代码只调一个入口：

```python
import litellm
resp = litellm.completion(
    model=settings.llm_model,      # 改这一行就换模型
    messages=messages,
    tools=tools,
    tool_choice="auto",
)
```

切换模型只需在 `.env` 改一行：


### 7.4 LLMClient 适配层（新增模块）

```python
# server/llm.py（草图）
import litellm

class LLMClient:
    def __init__(self, settings):
        self.model = settings.llm_model
        self.api_base = settings.llm_api_base   # 可选,自部署 / 私有化用
        self.api_key  = settings.llm_api_key    # 可选,覆盖环境变量
        self.timeout  = settings.llm_timeout
        litellm.drop_params = True               # 不支持的参数自动剥离
        litellm.set_verbose = False

    def complete(self, messages, tools=None, **kw):
        return litellm.completion(
            model=self.model,
            messages=messages,
            tools=tools,
            tool_choice="auto" if tools else None,
            api_base=self.api_base,
            api_key=self.api_key,
            timeout=self.timeout,
            **kw,
        )
```

后端所有 LLM 调用都经过此类，便于：监控、重试、降级、A/B 测试、模型路由。

| 阶段 | 策略 | 备注 |

### 7.6 必备能力清单（所有可选模型须满足）

- ✅ **OpenAI 兼容的 tools / tool_calls function calling**（核心要求）
- ✅ 可稳定调用 **≥ 8 个工具**（我们在 §9.4 规划了 8 个）
- ✅ Tool 调用循环 **≥ 10 轮**稳定（自由模式可能多次 tool→result→tool）
- ✅ 上下文窗口 **≥ 32K**（多轮 + 大 metadata + 工具结果会膨胀）
- ✅ 中文理解（业务用户中文提问、中文 SAP 字段名）
- ⚠️ 上线前必须在每个候选模型上跑 30 个用例的回归测试集

### 7.7 数据出境与合规

LLM 选型直接影响数据出境合规判定：

| 模型 | 提示词数据流向 | 适用场景 |
|---|---|---|
| Claude / GPT (官方 API) | 出公网到美国 | 非敏感业务，已签 DPA |
| Claude on Bedrock | 出公网到 AWS（区域可选） | 已部署在 AWS 的企业 |
| Qwen (Dashscope 杭州) | 不出境，但出企业网到阿里云 | 已采购阿里云的企业 |
| DeepSeek (官方) | 不出境，但出企业网 | 一般敏感度业务 |
| **本地部署 (Ollama / vLLM)** | **完全不出企业内网** | 涉密、金融、政企 |

**重要约束**：无论选哪家，遵循 §14 安全要点的「LLM 数据最小化」原则 —— LLM 只见样本数据（前 5 行 + 字段名），
完整数据走后端旁路。这一约束对所有候选模型生效，是产品级的设计护栏。

### 7.8 风险与限制

| 风险 | 缓解 |
|---|---|
| 不同模型 tool_calls 行为细微差异（参数缺失、类型偏差） | LiteLLM 抹平 95%；剩余在 `LLMClient` + 工具入参 schema 上做防御性校验 |
| 中小模型 tool 调用不稳定（漏调、参数格式错） | M3 阶段对每个候选模型跑回归集，**评分 < 80% 不允许做主力** |
| Reasoning 模型 (DeepSeek-R1 / QwQ) 输出 `reasoning_content` | LiteLLM 已规范化；超长 reasoning 截断后存审计，不送回上下文 |
| 国内模型并发限速严 (QPS 限制) | 后端令牌桶限流 + 队列 + 自动切备用模型 |
| 成本失控（用户狂问） | 审计表记录每次 token + 单价；按用户/部门设月度配额；达阈值告警 |
| 私有化模型能力不及 SaaS | 仅在涉密场景启用；非涉密走 SaaS 强模型 |
| LiteLLM 版本兼容性 | 锁定版本（`>=1.55,<2.0`），升级前回归测试 |

## 8. BW 数据层抽象 (Live / Mock 双实现)

### 8.1 设计动机

现实约束:
- **开发机不一定能连 SAP BW**（本机无 SAP / 内网未联通 / 客户环境不便频繁拨号）
- **UI/UX 迭代必须独立于 BW 后端**（前端开发时不应被 BW 数据可用性卡住）
- **演示与培训不能依赖真实数据**（防泄密 / 防误改 / 防业务波动）
- **回归测试需要确定性数据**（真实 BW 数据每天变，测不稳）
- **新员工 onboarding 5 分钟跑起来**（不必先拿到 SAP 账号）

**策略：把 BW 数据访问抽象成一个接口，提供 Mock 与 Live 两个实现，启动时按配置选择。**
**后期接 SAP 只需切配置 + 通过验收清单，业务代码（agent / orchestrator / excel builder / 前端）零改动。**

### 8.2 BWClient 接口契约

```python
# server/bw_interface.py
from typing import Protocol, Any
from dataclasses import dataclass

@dataclass
class ODataResponse:
    status_code: int
    url: str                       # mock 模式下用伪 URL,保留可读性
    json: Any | None
    text: str
    error: str | None = None

class BWClient(Protocol):
    """所有 BW 实现的统一契约。LiveBWClient 和 MockBWClient 都实现它。"""

    def list_services(self, search: str | None = None, top: int = 50) -> ODataResponse: ...

    def get_metadata(self, service: str) -> ODataResponse: ...

    def execute_query(
        self,
        service: str,
        entity_set: str,
        *,
        filter: str | None = None,
        select: str | None = None,
        orderby: str | None = None,
        top: int | None = 100,
        skip: int | None = None,
        expand: str | None = None,
        apply: str | None = None,
        count: bool = False,
    ) -> ODataResponse: ...
```

工厂选择实现:

```python
# server/bw_factory.py
def make_bw_client(settings: Settings) -> BWClient:
    if settings.bw_mode == "mock":
        return MockBWClient(data_dir=settings.mock_data_dir)
    if settings.bw_mode == "live":
        return LiveBWClient(settings)
    raise ValueError(f"未知 BW_MODE: {settings.bw_mode}（应为 mock 或 live）")
```

`.env` 切换：

```ini
BW_MODE=mock                    # 开发 / 测试 / 演示（启动期默认）
# BW_MODE=live                  # 联调 / 生产
MOCK_DATA_DIR=./mock_data
```

### 8.3 LiveBWClient （真实 BW）

复用现有 [bw_client.py](bw_client.py) 实现，封装：

- HTTP Basic Auth + `sap-client` + `sap-language`
- 服务目录 `/sap/opu/odata/iwfnd/CATALOGSERVICE;v=2/ServiceCollection`
- `$metadata` EDMX XML 拉取 + 简化解析
- `$filter / $select / $orderby / $top / $skip / $expand / $apply` 透传
- 错误归一化为 `ODataResponse`

**后期切到真实 BW 时不需要改这个文件**，只改 `.env` 与凭据配置。

### 8.4 MockBWClient （离线模拟）

完全离线，从本地 `mock_data/` 目录读取预制的服务清单 + 元数据 + 数据行。
对 OData 查询参数在内存中解释执行。

**实现要点**:

| 关注点 | 实现 |
|---|---|
| `list_services` | 读 `catalog.json`,按 `search` 过滤 `TechnicalServiceName/Title/Description` |
| `get_metadata` | 直接返回 `services/<service>/meta.json`，跳过 EDMX 解析 |
| `execute_query` | 加载 `services/<service>/data/<entity_set>.csv` 到 DataFrame，按 OData 参数过滤 / 投影 / 排序 / 切片 |
| 真实感 | 默认每查询 sleep 200-500ms 随机；可关闭 (`MOCK_LATENCY=0`) |
| 错误注入 | catalog 里某服务可标记 `"always_error": "503 Backend unavailable"`，测试降级路径 |
| 中文 label | meta.json 保留 SAP 中文 label，与真实 Gateway `sap-language=ZH` 行为一致 |
| 字段命名贴近真实 | 用 BW 现实风格 (`NETWR_F`, `/BIC/ZCUST`, `0CALMONTH`) 而非 `field1`、`name` |

## 9. Skills 子系统（核心创新）

### 9.1 设计灵感

借鉴 [Anthropic Skills](https://platform.claude.com/docs/en/agents-and-tools/skills) 的「渐进式披露」思想：
**每个 Skill 是一个文件夹，描述写在 `SKILL.md` 里**，LLM 先看简介，相关时再读详情，
进一步把数据源定义、Excel 模板、图表配置打包在一起，由 BW 管理员维护。

> 注：该思想由 Anthropic 提出，但本系统不绑定 Claude —— 任何支持 function calling 的 LLM
> (Qwen / DeepSeek / GPT) 都能按同样的方式调用 `list_skills` → `load_skill` → `run_skill`。

### 9.2 一个 Skill 长什么样

```
data/skills/
└── monthly_sales_region/                # skill_id
```

#### `SKILL.md` 示例

```markdown
---
id: monthly_sales_region
title: 月度销售大区简报
owner: BW 团队 - 张三
version: 3
keywords: [销售, 大区, 月报, 月度, region, sales]
visible_to: [sales, finance, exec]   # 角色组白名单
---

# 月度销售大区简报

## 适用场景
大区经理月初汇报当月业绩,需要按办事处 (Office) 看销售额、毛利、同比环比。

## 用户提供的参数
- `month`: 必填,YYYY-MM 格式,如 2026-05
- `region`: 必填,大区代码 (HD/HN/HB/HX),自然语言"华东"自动映射 HD
- `top_n`: 选填,默认 10,展示前几个办事处

## 给 LLM 的额外指引
- 销售额字段用 NETWR_F (净销售额),不要用 GROSS
- 同比环比的计算请用 OData $apply 一次性聚合,避免多次往返
- 输出表格列序: 排名 | 办事处 | 销售额 | 毛利 | 毛利率 | 同比 | 环比
```

#### `service.yaml` 示例

```yaml
service: ZBW_SALES_SRV
entity_set: SalesByOfficeView

# 参数 → OData filter 模板（用 jinja2 占位）
filter_template: |
  Region eq '{{ region }}' and YearMonth eq '{{ month | replace("-", "") }}'

select: [OfficeCode, OfficeName, NETWR_F, GROSS_PROFIT, GP_RATIO, YoY, MoM]
orderby: NETWR_F desc
top: "{{ top_n | default(10) }}"

# 后处理(可选): 用 pandas 表达式做派生列、排序、去重
post_process:
  - "df['Rank'] = range(1, len(df) + 1)"
```

#### `template.xlsx`

一个普通 .xlsx，第一行表头是占位符或固定列名；管理员可以用 openpyxl
预先做好千分位、冻结窗格、条件格式、图表区域。后端用 openpyxl 把数据
按列名填入指定 sheet。

## 10. Excel 生成

### 10.1 引擎选型

**主力**：[openpyxl](https://openpyxl.readthedocs.io/) —— 纯 Python、支持公式、条件格式、图表、冻结窗格、列宽自适应、合并单元格。

**不选**：
- `xlsxwriter`：单向写、不能读模板
- `pandas.to_excel`：不能保留模板的格式
- `xlwings`：要装 Excel COM，服务器跑不了

### 10.2 三种生成路径

| 路径 | 触发场景 | 实现 |
|---|---|---|
| **A. 模板填充** | Skill 提供了 `template.xlsx` | 用 openpyxl 打开模板 → 定位数据区（按 sheet 名 + 起始单元格）→ 写入 DataFrame → 保存 |
| **B. 默认模板** | 自由模式，无 Skill 模板 | 启动时预制 `default_template.xlsx`（一个简洁的浅色风格） → 同上 |
| **C. 多 sheet 合成** | 对比类查询（"三个季度"） | 一个 workbook 多 sheet，每 sheet 一段数据，再加一个汇总 sheet |

### 10.3 Excel 标准格式约定

每个产出的 Excel 至少含 2 个 sheet：

**Sheet 1 · `数据`**
- A1:An 表头（粗体 + 灰底）
- 数字列右对齐，千分位 `#,##0.00`
- 日期列 `yyyy-mm-dd`
- 冻结首行 + 首列
- 列宽根据内容自适应

**Sheet 2 · `查询信息`**（审计冗余）
- 提问原文
- 命中的 Skill / 版本
- 实际跑的 OData URL
- 查询时间
- 数据行数
- 用户、用时、LLM 模型与 token 消耗

可选 **Sheet 3+ · 图表**（按 Skill 配置）

### 10.4 文件命名

`{skill_id 或 free}_{yyyymmdd_HHMMSS}_{username}.xlsx`

例如：`monthly_sales_region_20260602_143215_zhangsan.xlsx`

### 10.5 大数据量处理

| 行数 | 策略 |
|---|---|
| < 1000 | 直接全量进 Excel |
| 1000-50000 | openpyxl `write_only` 模式（流式写，低内存） |
| 50000-200000 | 走 streaming，但提醒用户「数据较多，建议加筛选」 |
| > 200000 | 拒绝，让用户加筛选；产品定位是分析不是 dump |

## 14. 安全与合规要点（仅列与 v0.1 不同处）

1. **Skills 内容由管理员维护，需注入防护**：
   - `service.yaml` 里的 `filter_template` 用 jinja2 沙箱（`SandboxedEnvironment`），禁止访问 Python 内置
   - 上传的 `template.xlsx` 扫描宏（含 VBA 直接拒绝）
   - 上线前管理员双人复核（界面提供「待审核」状态）
2. **Excel 输出脱敏**：
   - 后端在写 Excel 前比对 `sensitive_fields`，匹配则按 `mask_mode` 替换（`redact` → `***`、`partial` → 保留首尾、`hash` → SHA256 前 8 位）
3. **LLM 数据最小化**：
   - LLM **不见**完整数据，只见每个 EntitySet **前 5 行样本** + 总行数
   - 完整数据走 `BWClient` → `ExcelBuilder` 直通
   - 这一约束对所有候选 LLM（包括国内/本地模型）一视同仁，避免敏感数据外泄
4. **Skill 的可见性控制**：
   - `visible_to` 配合用户的 `groups`，画廊只展示有权限的 Skill；LLM 调 `list_skills` 时也按角色过滤
